MinkaDE · shell architecture proposal

One socket, two runtimes.
Each pixel owned by the engine that renders it best.

The Minka shell splits along a performance boundary: Quickshell (Qt6/QML) owns the widget- and service-heavy surfaces, Guido (Rust + wgpu) owns the latency-critical animated overlays. Both are ordinary peers on ShojiWM's existing NDJSON IPC socket — the compositor contract doesn't change at all.

Topology

Process layout

ShojiWM's TypeScript config runtime already hosts a Unix socket that broadcasts reactive state and accepts commands. shoji-bar-2 proved the model with one GTK client; the hybrid simply connects two clients instead. Broadcasts fan out to every connection, so neither process needs to know the other exists.

ShojiWM

Compositor · Rust / Smithay

Owns input, layout, DRM/HDR, blur & liquid-glass shaders. Its Node config runtime hosts the IPC server and wakes the Rust side via SIGUSR1 after each mutation.

$XDG_RUNTIME_DIR/shojiwm-$WAYLAND_DISPLAY.sock
newline-delimited JSON · requests with id-correlated responses · fan-out broadcasts

minka-shell

Quickshell · Qt6 / QML
  • Scene-graph rendering, rich text, vector shapes
  • Built-in daemons: StatusNotifier tray, notification server, PipeWire, UPower, MPRIS, NetworkManager
  • Socket I/O async on the Qt event loop; QML renders on a separate thread

minka-fx

Guido · Rust / wgpu
  • SDF shapes, spring-physics transforms, per-frame animation
  • Socket registered as a calloop source — same loop as rendering, never blocking it
  • Same language as the compositor; IPC payload types shared via a small minka-ipc crate
Ownership

Who draws what

The split follows each runtime's strengths — and Guido's current gaps (v0.5: no text input, no tray, no popup system) make the boundary crisp: anything with forms, lists, or system services is Quickshell; anything transient and animation-bound is Guido.

IPC

The contract stays, the exec channel goes

The wire format and method surface are exactly what shoji-bar-2 already speaks. The one structural change: today keybindings spawn ags request start-menu toggle — a process exec per keypress. The shells hold persistent connections, so the config broadcasts ui.* events instead. Zero spawns, one less CLI dependency. 

// requests (client → server, id-correlated)
{ "id": 1, "method": "workspaces.get" }
{ "method": "workspaces.activate",
  "params": { "monitor": "DP-1", "index": 2 } }
{ "method": "windows.activate",
  "params": { "windowId": "…" } }
// broadcasts (server → all clients)
{ "event": "workspaces.changed", "payload": … }
{ "event": "snap.preview",
  "payload": { "monitor", "rect", "kind" } }
{ "event": "dock.proximity", "payload": … }
// new: replaces `ags request` execs
{ "event": "ui.startMenu",
  "payload": { "connector", "action" } }
R1
Never block a render loop on the socket. Quickshell: async Quickshell.Io.Socket on the Qt event loop. Guido: the socket is a calloop event source beside the frame callback. No thread ever issues a synchronous read.
R2
Diff and coalesce at the source. The config runtime already JSON-diffs workspaces.changed and coalesces mutations per tick — every new event keeps that discipline, so idle traffic is zero.
R3
Transmit state, not frames. Broadcasts carry targets (a snap rect, a proximity bool). Clients animate toward them locally. Bandwidth stays trivial and JSON stays fast enough that a binary protocol is not worth its complexity.
R4
Reconnect is a feature. The socket is recreated on config hot-reload; both clients auto-reconnect and re-fetch workspaces.get, exactly as shoji-bar-2 does today.
Motion

Why 20 Hz of IPC looks like 165 Hz

The hollow rectangle is what the compositor actually sends — a snap target that jumps a few times per second. The red rectangle is Guido's spring integrating toward it every frame. Smoothness is manufactured client-side, so IPC rate never dictates animation quality.

Duo mode

Zenbook Duo placement

A ShellLayout policy (shared as config both processes read) resolves monitor roles at startup and on hotplug: when the 1920×515 ScreenPad is present, every persistent surface pins there and the main panel stays clear; otherwise the single-screen KDE-style layout applies. Guido overlays follow the window they serve, so they appear on whichever output the drag is on.

Milestones

  1. Skeleton Quickshell scaffold + ShojiIpc QML singleton speaking the existing protocol; bare bar with live workspaces on both displays. shoji-bar-2 keeps running as the reference until parity.
  2. Widget parity Port the shoji-bar-2 tree — dock, start menu, status/clock menus, tray, notifications — using caelestia-shell's structure as the layout reference.
  3. minka-fx Guido process + minka-ipc crate; take over snap preview and dock reveal; add zephyr-style OSDs. Retire the ags request exec path for ui.* broadcasts.
  4. Identity pass Eternal-Darkness theme tokens shared by both runtimes (one theme.json, hot-reloaded via a theme.changed broadcast); compositor-side blur flag for shell surfaces.
Known risk: Guido is v0.5 with a tiny community — treat it as an implementation detail behind the minka-fx process boundary. Everything it renders is transient and inputless, so if it stalls as a dependency, the fallback is a few hundred lines of smithay-client-toolkit + wgpu we write ourselves (same crates, same event loop model), not a rewrite of the shell.
